WSN任务2学习记录

Chapter 1 AI communication

Q1 现在进入恶意节点检测优化阶段，首先如果让你设计你会如何设计?

多维度特征工程 特征维度（~10维）:

F2/F3/F4（时序特征）：恶意节点可能间歇性丢包（突变为0再恢复），而正常节点波动平缓。单纯的累积率抹平了这种模式。 F7（中介中心性）：高价值路由节点更可能被策反，但也承担更多转发，需要在上下文中判断。 F8（IN置信度）：IN距离RN越远、链路越差，能观测到的包越少，该观测的可信度越低。这为后续加权提供依据。


Q2 只对IN可观测的RN节点做检测是什么意思，恶意节点不在其中分布该如何检测出?

IN提供的是”置信度权重”，不是”检测边界”。

IN监控的RN → 样本量充足（中转大量包），方差低 → 高置信判决 纯EN（无IN）→ 每轮仅20包，方差高但多轮可累积 → 稍低置信但仍有足够数据 既非EN也非RN → 零观测，不可能评估 那IN机制到底带来了什么 回到你的原问题——IN给转发率检测带来的增量价值：

维度 无IN（当前任务六） 有IN
检测范围 所有received>0的节点 相同（不缩小）
样本量 RN>1000包，纯EN≈4000包（200轮×20） 相同
置信度权重 无 IN信誉分可作为该观测的可信度
空间定位 无 知道哪个IN在监视哪个RN，可追责
新特征 无 IN-TO-RN链路质量、IN自身剩余能量等
交叉验证 无 多个节点可能同时观测同一RN的转发

IN没有减少可检测节点数，而是给同一份转发率数据附加了元信息——谁观测的、链路质量如何、观测者是否可信。 好，让我先完整理解当前数据流和需要改动的点，然后给出实现方案。

  1. task5 — 不再丢弃 reputation_scores，存入 ForwardingRateResult
  2. task6_kmeans（共享模块）— 每轮累加 monitored_rn_stats 中的IN监控数据，新增 compute_in_features() 产出置信度权重
  3. task6_dpc — 自动继承，DPC密度计算用IN置信度加权
  4. CNN/LSTM/GNN/TGCN — 各自的序列累加函数同样改动，IN特征作为额外输入通道
  5. 范围不变 — 所有有转发数据的节点都参与检测，IN只是提供权重

Here is Claude’s plan:

IN机制集成到恶意节点检测 — 实现方案

目标

将IN监控机制的元信息（信誉分、监控关系）作为置信度权重和附加特征引入检测管线，不缩小检测范围。

当前丢弃的IN数据

丢弃数据 位置 用途
monitored_rn_stats（每轮） ForwardingRateResult 内 task6内循环忽略
reputation_scores MonitoringResult 内 task5 simulate_round 用 _ 丢弃
monitoring_node_indices 存入 MultiRound*Result 从未被任何检测逻辑读取

实现步骤

Step 1: task5 — 传递IN信誉分

文件: task5_forwarding_rate_monitoring.py

  • ForwardingRateResult 新增字段 in_reputation_scores: np.ndarray
  • simulate_round() 不再丢弃 monitoring_result，提取 reputation_scores 存入 result
  • 行 100: nodes, _, monitoring_result, routing_paths = ... → 保留 monitoring_result
  • 行 190 附近: result 构造时传入 reputation_scores

Step 2: task6_kmeans — IN累积 + 特征提取

文件: task6_kmeans_malicious_detection.py（DPC复用此文件函数）

2a. MultiRoundObservationResult 新增字段:

2b. simulate_multi_round_observation() 修改: - 首轮：从 first_round_result.monitored_rn_stats 初始化累加器 - 每轮内循环：解析 round_result.monitored_rn_stats，为每个被监控RN累加 IN监控轮次 + IN信誉分 - 构造 observation_result 时填充新字段

2c. 新增 compute_in_features(observation_result):

- rounds_with_data = node_received_counts > 0 的掩码对应轮次数（近似用总轮数 × 收到包比例） - norm_reputation = reputation_sum / max(1, monitored_rounds) 归一化

Step 3: task6_kmeans — 检测算法使用IN特征

3a. K-means：将 in_confidence 作为样本权重引入加权1D k-means（质心更新时按权重加权平均）

3b. 判决阶段：节点的 in_monitored_ratio 作为”观测可靠度” — 高可靠且高置信的判决优先采纳，低可靠节点标记为 suspicious 而非直接定论

Step 4: task6_dpc — 复用kmeans改动

文件: task6_dpc_malicious_detection.py

  • 自动继承 Step 2 的 simulate_multi_round_observation() 和 compute_in_features()（已导入）
  • DPC局部密度计算时，用 in_confidence 加权：密度 = Σ(in_confidence[j] * kernel(distance))
  • 聚类分配时融入 IN 特征距离

Step 5: task6_cnn/lstm/gnn/tgcn — IN序列特征

文件: task6_cnn_malicious_detection.py, task6_lstm_malicious_detection.py, task6_gnn_malicious_detection.py, task6_tgcn_malicious_detection.py

每个文件的 simulate_multi_round_sequences() 做相同改动： - 首轮提取 monitored_rn_stats 初始化 - 内循环每轮累加 IN 数据到 MultiRoundSequenceObservationResult - MultiRoundSequenceObservationResult 新增同样的 3 个 IN 字段 - 各自的 compute_observed_rate() 或等价函数也支持 IN 特征

检测模型改动： - CNN/LSTM: 输入从 (rounds, 1) 扩为 (rounds, 2) — 加 in_confidence_per_round 通道 - GNN/TGCN: 节点特征矩阵增加 IN 特征列

Step 6: 验证

检查: 1. in_confidence 值域在 [0,1] 且 IN 监控节点值更高 2. miss rate < 3%, false-positive rate < 5% 指标不退化 3. 对比修改前后的准确率、召回率

关键原则

  • 范围不变: 所有 received > 0 的节点仍然参与检测
  • IN做增强: 有IN监控 → 高权重高置信；无IN → 低权重但仍有数据支撑
  • 向后兼容: 不改变 simulate_round() 的返回签名（只增加 ForwardingRateResult 字段）

好问题。让我分析恶意节点成为RN的情况。

恶意节点作为RN的场景

当恶意节点恰好被选为RN时：

维度 恶意RN 恶意纯EN 恶意非路由节点
包量 极高（中转大量包） 低（仅20包/轮） 零
IN监控 有（task3强制每个RN分配IN） 无 无
统计置信度 最高 低 不可检测
检测难度 最容易 中等 不可能

恶意RN反倒是最容易被检测到的——它转发的包多，IN盯着它的样本量最大，统计上最容易暴露异常的转发率。

但有一个关键问题

任务五中 assign_forwarding_rates() 是全网随机分配恶意节点的（task5 line 86），与路由角色无关。恶意节点成为RN纯粹是随机的。而我方案中的IN置信度权重会让有IN监控的节点判决权重更高，这恰好与”恶意RN最容易被检测”吻合——设计是自洽的。

有一个边界情况需要考虑

恶意节点能否成为IN本身？如果IN是恶意的，它监控的数据还可信吗？

当前实现中IN选择基于 reputation_scores（能量+度+近sink），恶意性是独立随机分配的，所以恶意节点可能被选为IN。但当前系统中IN并不独立”报告”数据——IN统计同样来自全局计数器，所以IN的恶意性不影响观测数据本身。不过这是架构上的一个隐患。

Q3 现在可使用哪些特征，解释这些特征的实际含义

特征 实际含义 一般异常信号
observed_rate / cumulative_forwarding_rate 节点累计转发率 = 转发包数 / 接收包数 明显偏低常表示选择性转发
rate_seq_mean 多轮转发率序列的平均水平 长期偏低更可疑
rate_trend 转发率趋势（末轮 - 首轮） 持续下降可能是攻击逐步加重
rate_std 转发率波动大小 过大可能表示不稳定或间歇攻击
fulfillment_ratio 实际转发量 / 期望转发量 明显 < 1 说明“该转没转够”
neighbor_rate_mean 邻居节点平均转发率 作为本地基线
neighbor_rate_std 邻居转发率离散度 小方差下自身偏离更可信
neighbor_deviation 自身转发率 - 邻居均值 显著为负时可疑
role_zscore 相对“同角色”节点的标准化偏差 负值很大说明该角色里异常差
role_id 节点角色（EN/RN/IN/backbone/ordinary） 不同角色阈值应不同
participation_log 参与路由/事件次数（log后） 参与越多，统计越稳定
hop_norm 到 sink 的归一化跳数 反映位置与转发压力差异
degree_norm 归一化邻居数（连通度） 低度数节点天然更易波动
in_confidence IN监测置信度（覆盖率×信誉） 低时该样本结论应降权
in_monitored_ratio 被IN监测轮次占比 低覆盖意味着观测不充分
